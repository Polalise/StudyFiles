# 18. 당뇨병 예측 프로젝트 포트폴리오 문서 만들기

## 이번 장의 목표

13장부터 17장까지 우리는 당뇨병 위험 예측 프로젝트를 단계적으로 만들었다.

이번 장의 목표는 다음 한 문장이다.

> 지금까지 만든 분석과 모델을 포트폴리오에서 설명 가능한 프로젝트 문서로 정리한다.

프로젝트는 코드만 있다고 완성되는 것이 아니다.

취업용 포트폴리오에서는 다음을 설명할 수 있어야 한다.

```text
무슨 문제를 풀었는가?
어떤 데이터를 사용했는가?
어떤 전처리를 했는가?
왜 그 모델을 사용했는가?
어떤 지표로 평가했는가?
모델이 어떤 한계를 가지는가?
다음 개선 방향은 무엇인가?
```

## 프로젝트 제목 정하기

제목은 너무 거창하지 않게, 하지만 목적이 보이게 적는다.

좋은 예:

```text
건강 지표 기반 당뇨병 위험 예측 MLP 프로젝트
```

피해야 할 예:

```text
AI 당뇨병 진단 시스템
```

이 프로젝트는 의학적 진단 시스템이 아니다. 교육용 데이터 기반 위험 예측 프로젝트다.

## 문제 정의

문제 정의는 프로젝트의 시작점이다.

이 프로젝트의 문제 정의는 다음처럼 쓸 수 있다.

```text
BRFSS 건강 지표 데이터를 사용해 당뇨병 여부를 이진 분류하는 MLP 모델을 구축한다.
모델의 목적은 의학적 진단이 아니라, 딥러닝 기반 건강 위험 예측 흐름을 학습하고 포트폴리오화하는 것이다.
```

여기서 중요한 단어는 “이진 분류”와 “위험 예측”이다.

## 데이터 설명

데이터 설명에는 다음이 들어가야 한다.

```text
데이터 파일
행과 열의 의미
타깃 컬럼
주요 입력 특성
클래스 분포
```

예시:

```text
데이터는 BRFSS 2015 건강 지표 기반 당뇨병 이진 분류 데이터이다.
한 행은 한 사람의 건강 설문 및 지표를 나타내며,
입력 특성에는 고혈압 여부, 고콜레스테롤 여부, BMI, 흡연 여부, 신체활동, 나이 구간 등이 포함된다.
타깃은 Diabetes_binary이며, 0은 당뇨병 없음, 1은 당뇨병 있음을 의미한다.
```

## 전처리 설명

전처리는 모델이 데이터를 제대로 받을 수 있게 만드는 과정이다.

이 프로젝트에서는 다음을 설명할 수 있다.

```text
입력 X와 정답 y 분리
train/validation split
stratify로 클래스 비율 유지
StandardScaler 적용
scaler는 학습 데이터에만 fit
검증 데이터에는 transform만 적용
```

특히 scaler 부분은 중요하다.

```text
데이터 누출을 막기 위해 StandardScaler는 학습 데이터에만 fit하고,
검증 데이터에는 학습 데이터 기준으로 transform만 적용했다.
```

이 문장은 프로젝트 품질을 꽤 올려준다.

## 모델 설명

모델 설명은 너무 복잡하게 쓸 필요 없다.

초보 포트폴리오에서는 구조와 선택 이유가 명확하면 좋다.

```text
표 형태의 건강 지표 데이터이므로 Dense 층 기반의 MLP를 사용했다.
출력층은 이진 분류를 위해 sigmoid 활성화 함수를 사용했으며,
손실 함수는 binary_crossentropy를 사용했다.
과적합을 줄이기 위해 Dropout과 EarlyStopping을 적용했다.
```

## 평가 설명

평가에서는 accuracy만 쓰지 않는다.

건강 위험 예측에서는 precision과 recall이 중요하다.

```text
accuracy는 전체 예측 중 맞힌 비율을 보여준다.
precision은 위험이라고 예측한 사람 중 실제 위험인 비율을 보여준다.
recall은 실제 위험인 사람 중 모델이 찾아낸 비율을 보여준다.
f1-score는 precision과 recall의 균형을 보여준다.
```

그리고 threshold 이야기를 넣으면 좋다.

```text
sigmoid 출력은 0~1 사이의 예측 점수이며,
threshold를 조정하면 precision과 recall의 균형이 달라진다.
건강 위험 예측에서는 FN을 줄이는 것이 중요할 수 있으므로 threshold 선택이 프로젝트 목적에 따라 달라질 수 있다.
```

## 해석 설명

16장에서 배운 해석 내용을 넣는다.

```text
Permutation Importance를 사용해 각 특성을 섞었을 때 검증 F1 점수가 얼마나 하락하는지 확인했다.
이를 통해 모델이 어떤 입력 특성에 민감하게 반응하는지 관찰했다.
단, 중요도는 인과관계가 아니라 모델의 예측 의존도로 해석했다.
```

마지막 문장이 특히 중요하다.

## 재사용 가능한 예측 흐름

17장에서 만든 예측 함수도 포트폴리오에 넣을 수 있다.

```text
학습된 Keras 모델뿐 아니라 StandardScaler와 feature_names를 함께 저장했다.
새 입력값이 들어오면 학습 때와 같은 컬럼 순서와 같은 scaler를 적용한 뒤 예측하도록 함수화했다.
이를 통해 노트북 분석을 데모 앱으로 확장할 수 있는 구조를 만들었다.
```

## 한계

좋은 포트폴리오는 한계를 숨기지 않는다.

이 프로젝트의 한계는 다음처럼 적을 수 있다.

```text
이 모델은 의료 진단 도구가 아니다.
설문 기반 데이터이므로 실제 임상 검사 데이터와 다르다.
특성 중요도는 인과관계를 의미하지 않는다.
데이터 출처와 수집 방식에 따른 편향 가능성이 있다.
실제 서비스에는 전문가 검토와 추가 검증이 필요하다.
```

한계를 적는 것은 약점이 아니라 성숙한 태도다.
